# Readme

## 기본 공통 정보를 기반으로 하는 컨트롤러

이 문서는 기본 공통 정보를 기반으로 하는 컨트롤러의 목표와 주의사항을 정의합니다. 이 컨트롤러는 모든 컨트롤러를 담당하지 않으며, 일부만을 다루도록 설계되었습니다. 특히 공통 정보와 상위 레이어의 정보를 중점적으로 처리합니다.

## 컨트롤러 목표

1. **일부 컨트롤러만 다루기**: 모든 컨트롤러를 담당하지 않고, 일부만을 사용할 것으로 가정합니다. 특히 상위 레이어의 공통 정보를 다루는 것을 목표로 합니다.
2. **DTO 추가 상황 대비**: DTO가 갑자기 추가되는 상황도 고려하여 유연하게 대처할 수 있도록 합니다.
3. **엔티티 리턴 제한**: 엔티티를 직접 반환하는 경우는 매우 한정적으로 사용해야 합니다.
4. **레포지토리 사용 금지**: 다른 서비스를 참조할 수는 있지만, 컨트롤러에 직접 레포지토리를 사용하지 않습니다.
5. **서비스 종속성 확인**: 다른 서비스를 참조할 때, 해당 서비스가 다른 서비스에 종속적인지 확인 후 사용하며, 종속적이라면 서비스 단위에서 이를 처리합니다.
6. **복잡한 정보 분리 가능성 고려**: 기준 정보만 작업하며, 다른 정보가 복잡할 경우 별도 프로젝트로 분리할 수 있습니다. 서비스 영역의 통신 케이스를 고려해 설계합니다.

## 제네릭 파라미터 설명

- `<D>`: 기본적인 엔티티의 DTO
- `<REQ>`: 요청 DTO의 형식
- `<RES>`: 응답 DTO의 형식
- `<T>`: 객체 자체를 다루는 경우에 사용할 제네릭
- `<INSERT>`: 입력용 DTO

## 컨트롤러 작업 시 주의사항

1. **컨트롤러 수정 금지**: 이미 만들어진 컨트롤러는 수정하지 않고, 필요 시 새로 추가하는 방식으로 작성합니다.
2. **요청-응답 DTO 변경 금지**: 요청-응답 DTO 자체는 함부로 변경하지 않습니다.
3. **새로운 DTO 추가 시 오버라이드 사용 지양**: 새로운 DTO를 추가할 때는 오버라이드를 피하고, 독자적인 메서드를 사용합니다.
4. **객체 직접 처리 금지**: 컨트롤러에서 객체를 직접 처리하지 말고 서비스 영역으로 넘깁니다.
5. **모든 컨트롤러는 리턴 필요**: 모든 컨트롤러에는 반드시 리턴이 있어야 합니다.
6. **에러 표현 정의**: 에러 표현을 명확하게 정의해야 합니다.
7. **시큐리티 필터의 스코프 고려**: 시큐리티 필터에 정의된 스코프를 준수하며, 스코프를 위반하는 데이터 사용에 주의해야 합니다.
8. **메서드 설명 제공**: 각 메서드에 자체 설명을 넣으며, 인터페이스 오버라이드 시에는 기본 설명을 제공합니다.
9. **Mocking 테스트 사용**: 단위 테스트 시 mocking 테스트를 활용합니다.
